# 提示词创建助手

## 快速开始

创建意图识别提示词时，按以下步骤进行：

1. **明确识别场景** - 确定要识别什么意图，有多少个类别
2. **定义角色** - 明确AI的专业定位和背景
3. **描述任务说明** - 说明输入输出和判断规则
4. **设定任务目标** - 阐述达成的业务目标
5. **规范输出格式** - 定义JSON或其他格式标准
6. **列出禁止事项** - 明确不允许的行为

---

## 模板结构

### 1. 角色定义

定义AI的角色和专业背景。

**格式：**
```markdown
## 角色定义
你是一名【<专业角色>】，专注于在【场景】中，<核心能力>。

当前对话背景为：
> <业务上下文描述>

用户的回复用于表达：
- <用户意图1>
- <用户意图2>
```

**写作要点：**
- 角色要具体，如"意图识别专家"、"文本分类专家"
- 明确应用场景和业务背景
- 说明用户输入的用途和目的

**示例：**
```markdown
## 角色定义
你是一名【意图识别专家】，专注于在宽带报障场景中，识别用户对地址确认问题的真实意图。

当前对话背景为：
> 机器人已向用户确认地址，例如：**"xxxx，这个地址对吗？"**

用户的回复用于表达：
- 地址是否正确
- 或者表达不确定/未理解
```

---

### 2. 任务说明

详细描述具体的识别任务和判断规则。

**格式：**
```markdown
## 你的任务
根据用户的【输入内容】，准确判断其意图类型，并按指定格式输出结果。

---

## 意图分类说明

### <类别名称>
用户明确表示：
- <特征1>
- <特征2>

#### 常见表达（示例）
- "示例1"
- "示例2"

---
```

**写作要点：**
- 任务说明要清晰，说明输入和输出
- 每个类别要明确特征
- 提供足够的示例覆盖常见表达
- 示例要真实、具体

**示例：**
```markdown
## 你的任务
根据用户的【单轮回复内容】，准确判断其意图类型，并按指定 JSON 格式输出结果。

---

## 意图分类说明

### 肯定意图（affirm）
用户明确表示：
- 地址是正确的
- 地址没问题
- 可以继续报障

#### 常见表达（示例）
- "对的"
- "没错"
- "是这个"
- "对"
```

---

### 3. 任务目标

说明达成该意图识别的业务目标和期望效果。

**格式：**
```markdown
## 任务目标
<业务目标说明>

期望达成效果：
- <效果1>
- <效果2>
```

**写作要点：**
- 说明为什么要进行这个识别
- 描述识别成功后的业务价值
- 列出关键的成功指标

**示例：**
```markdown
## 任务目标
准确判断用户对地址确认的反馈，决定后续报障流程的走向。

期望达成效果：
- 肯定时，直接进入报障流程
- 否定时，引导用户提供正确地址
- 不明确时，重新发起确认询问
```

---

### 4. 输出格式

严格定义输出的格式规范。

**格式：**
```markdown
## 输出格式（必须严格一致）

### <场景1>
```json
{
  "字段名": "值",
  "字段名": "值"
}
```

### <场景2>
```json
{
  "字段名": "值"
}
```
```

**写作要点：**
- 为每个意图类别提供对应的输出格式
- 使用JSON格式便于程序解析
- 字段命名要清晰、一致
- 包含必要的话术字段（如answer）

**示例：**
```markdown
## 输出格式（必须严格一致）

### 用户明确肯定
```json
{
  "intent": "affirm",
  "answer": ""
}
```

### 用户明确否定
```json
{
  "intent": "deny",
  "answer": "好的，请您重新提供一下要报障的宽带地址"
}
```

### 无法判断（需要重新询问）
```json
{
  "intent": "unclear",
  "answer": "不好意思我没理解您的意思，请问您要报障的地址是{{ need_verify_address }}对吗？"
}
```
```

### 5. 禁止事项

列出不允许的行为，确保输出规范。

**格式：**
```markdown
## 禁止事项
不输出分析过程
不输出除指定格式以外的任何文本
不自行修改话术内容
不合并多个意图
不进行上下文推测
```

**写作要点：**
- 列出所有不允许的行为
- 使用简洁的否定句式
- 覆盖常见的输出错误场景

**示例：**
```markdown
## 禁止事项
不输出分析过程
不输出除 JSON 以外的任何文本
不自行修改确认话术内容
不合并多个意图
```

## 最佳实践

1. **示例要丰富** - 每个类别提供5-10个真实示例，覆盖不同表达方式
2. **边界要清晰** - 明确区分各个类别的特征，避免模糊地带
3. **格式要严格** - 输出格式必须与程序解析逻辑完全一致
4. **话术要专业** - 答复话术要符合业务场景和品牌调性
5. **测试要充分** - 使用边缘案例测试提示词的准确性

## 常见问题

**Q: 如何判断该分到哪个类别？**
A: 优先看用户表达的核心意图，而不是具体用词。多个特征符合时，选择最明确的那个。

**Q: 示例多少个合适？**
A: 建议每个类别至少5个示例，总共不少于15个示例，覆盖不同表达方式。

**Q: 输出格式可以是其他格式吗？**
A: 建议使用JSON格式，便于程序解析和扩展。如需其他格式，请确保程序端能正确处理。

## 创作流程

1. 与用户沟通，明确识别场景和类别
2. 按模板结构逐部分编写
3. 提供足够的示例
4. 验证输出格式是否符合要求
5. 迭代优化，提升准确性
